Pixel perfect rendering

Summary

In this blog post I will describe how you can draw pixel perfect graphics using OpenGL, to achieve a result similar to this:

We will achieve this by first rendering the game world into a small framebuffer with only the necessary pixels, and then rendering this framebuffer onto the screen with a slight offset, such that the camera appears to be exactly where it needs to be.

Rendering into pixel perfect framebuffer

First, we need to decide on the resolution to render the game at. In my case, I’ve taken the screen resolution and calculated how many pixels can fit, if every pixel occupied 4 screen pixels. This way, the game is able to support multiple resolutions with little change to the code.

Another option you could choose is to determine the size by how many tiles you want to render. This does not always work, as the number of pixels calculated rarely evenly divides the rendering screen, resulting in some artifacts.

Why tile based height doesn’t work

Here is an example:

Suppose you want to render 16x16 tiles on a 16x9 screen. For this example, let’s use 1920x1080 as the resolution we are rendering onto. Here is the table of how many screen pixels are needed for each game pixel:

There are no even divisions, which means that if you calculate the size of the intermediate framebuffer by the number of tiles, you will run into the problem that some pixels will take more space than others. An easy visualization is to try and fit 3 game world pixels into a screen of size 4. One of those pixels will be duplicated. On the other hand, if you draw a game world of size 6 on a screen of size 4, some pixels may stop being visible.

Scaling

Let’s try scaling them. In this case, we must decide how many screen pixels our game pixels must occupy. With a scaling of 2x, each game pixel occupies 2 screen pixels. With a scaling of 3x, it’s 3 screen pixels, and so on. We can calculate the size of our framebuffer by dividing the resolution of our screen by our scaling factor, and taking the ceiling of those numbers.

Borders

After you have the game world resolution calculated, you will want to add 2 to both the width and the height. This will be useful when rendering the framebuffer into the screen.

Rendering pixel art in OpenGL

OpenGL wasn’t created for pixel art games, but we can sure use it that way. You just need to make sure every texture you draw is aligned to the pixels exactly and your camera is offset by half a pixel when rendering.

Assuming your tiles are unit sized, and you know how many pixels are in each tile (let’s use 16 for this example), you can align your coordinates using this formula:

glm::vec2 result = glm::floor(position * 16.0f) / 16.0f;

Then, for the camera:

glm::vec2 snap_pos = glm::floor(position * 16.0f) / 16.0f;
glm::vec2 camera_pos = snap_pos + glm::vec2(0.5f) / 16.0f;
glm::vec2 pixel_offset = position - snap_pos;

You can either add half a pixel, or subtract half a pixel. The point here is to avoid aproximation errors in the shaders, so preparing the scene such that each texil lands right in the middle of a pixel is what’s important.

Another very important point here is that you save how much the camera was offsetted by the alignment. We will use this in the next section, when aligning the resulting pixel art into the screen.

Rendering to the screen

When rendering to the screen, draw a quad, but instead of using (0.0f, 0.0f) and (1.0f, 1.0f) for the texture coordinates (drawing the whole texture), we will only draw a part of it.

glm::vec2 pixel_offset = camera->getPixelOffset();

// (0, 0) is the top left corner of the screen. If your camera has it's
// coordinates in the game world, invert the y axis.
pixel_offset.y = 1.0f - pixel_offset.y;

// Transform pixel offset into [0.0, 1.0] range.
glm::vec2 pos = pixel_offset / game_world_resolution;

// Snap to screen pixels
pos = glm::floor(pos * screenRes) / screenRes;

// How many pixels fit on screen
glm::vec2 sz = screenRes / scaling;

// Transform into [0.0, 1.0] range
sz /= game_world_resolution;

// Compute the final texture coordinates.
glm::vec2 corner = pos + sz;
glm::vec4 coords(pos.x, pos.y, corner.x, corner.y);

GLfloat vertices[] = {
	 1.0f, -1.0f, coords[2], coords[3],  // bottom-right
	 1.0f,  1.0f, coords[2], coords[1],  // top-right
	-1.0f,  1.0f, coords[0], coords[1],  // top-left
	-1.0f, -1.0f, coords[0], coords[3],  // bottom-left
};

The code above is what renders the pixel art into the screen, taking into account how much offset we had to sacrifice in order to align it, and adding it back in, so long as the screen allows it.
